      Chain-of-Thought

      A reasoning technique that instructs the model to produce explicit intermediate steps before arriving at a final answer. Substantially improves accuracy on multi-step arithmetic, logic, and planning tasks.

      reasoning cot llm technique

      When to use

      Apply Chain-of-Thought whenever the task requires multi-step reasoning that a single-pass answer would likely compress or skip: mathematical word problems, logical deduction chains, SQL query construction from ambiguous specs, differential diagnosis, or any planning task where intermediate correctness matters.

      Skill template

      [Your standard role + context block here.]

## Task
[State the problem or question to solve.]

## Instructions
Think through this step by step before giving your final answer.
For each step:
1. State what you are about to determine.
2. Show your working or reasoning.
3. State the intermediate conclusion.

After all steps are complete, state your final answer clearly under the heading:
**Final answer:**

      How to apply it

      1. Anchor the role and context first. CoT does not replace a well-structured prompt — it augments it. Use the Intro to Prompting template as the base layer.
      2. Add the step-by-step instruction explicitly. The phrase “think step by step” is the minimal trigger. The full template above gives the model an output schema to follow, which produces cleaner, auditable reasoning.
      3. Ask for the final answer separately. The Final answer: heading creates a reliable extraction point so downstream parsing is not tangled with the reasoning trace.
      4. Verify intermediate steps, not just the conclusion. CoT’s value is in exposing the reasoning chain so you can identify exactly where a model goes wrong, not just that it did.

      When NOT to use

      • Simple lookup or classification tasks. CoT adds token cost and latency with no accuracy benefit when the answer is a direct fact retrieval.
      • Latency-sensitive pipelines. The extended output increases response time. For real-time UX, consider zero-shot or few-shot instead.
      • Tasks that require concise output. The reasoning trace will dominate the response; if downstream consumers only need the final answer, add an extraction step or use structured outputs.
