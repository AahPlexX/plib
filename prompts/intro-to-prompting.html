      RISEN — Universal Structured Prompt

      A production-grade, copy-paste-deployable prompt scaffold based on the RISEN framework (Role, Instructions, Steps, End-goal, Narrowing). Every field is deterministic — no ambiguous placeholders. Drop in your values and run.

      RISEN zero-shot structured universal role-prompting production

      Framework basis

      RISEN is a deterministic prompt scaffold derived from research on role prompting (Wang et al., 2023; Zheng et al., 2023), zero-shot instruction prompting (Wei et al., 2022), and output-format engineering (Schulhoff et al., 2025 — The Prompt Report). Each component maps directly to a specific failure mode it prevents:

      Role
      Constrains vocabulary, reasoning style, and assumed expertise level. A precise role reduces hallucination surface area and tones responses to the correct domain register. Source: role / persona prompting literature (Wang et al., 2023j; Reynolds & McDonell, 2021).
      Instructions
      The single imperative directive. One verb, one scope. Compound instructions produce compound errors. Source: directive component definition (Schulhoff et al., 2025 §1.2.1).
      Steps
      An ordered breakdown the model must follow. Explicit sequencing prevents steps from being compressed or reordered, which is the primary failure mode on multi-part tasks.
      End-goal
      States the success criterion explicitly. Models optimise for what is measurable in the prompt; without an explicit end-goal they optimise for fluency, not correctness.
      Narrowing
      Negative-space constraints (what NOT to do). Empirically more reliable than hoping the instruction implies the constraint. Source: alignment / ambiguity section (Schulhoff et al., 2025 §5.2.4).

      Model & task applicability

      • All frontier API-tier models (GPT-4 class, Claude 3+ class, Gemini 1.5+). Zero-shot instruction-following at this scale makes RISEN immediately effective without examples.
      • Not recommended below ~7B parameters without few-shot examples. Smaller models benefit from adding 2–3 worked exemplars after the Steps block. The instruction-following capability required for RISEN activates reliably only above ~70B parameters or with strong RLHF fine-tuning (Schulhoff et al., 2025; Tianpan, 2026).
      • Best for: code review, document drafting, data extraction, classification, structured analysis, technical explanation. For multi-step mathematical reasoning, augment with the Chain-of-Thought skill.

      Canonical template

      All-caps tokens are the only insertion points. Replace each one with a concrete value before sending. Do not leave any token as-is.

      ## Role
You are a EXPERT_ROLE with deep expertise in DOMAIN.
You communicate at a AUDIENCE_LEVEL level.

## Instructions
INSTRUCTION_VERB the following INPUT_TYPE and produce OUTPUT_TYPE.

## Steps
1. STEP_ONE — [what to do first]
2. STEP_TWO — [what to do second]
3. STEP_THREE — [what to do third]
4. If ambiguity is encountered at any step, state the assumption you are making
   before proceeding.

## End-goal
The output is successful when: SUCCESS_CRITERION.

## Narrowing
- Do NOT: CONSTRAINT_ONE
- Do NOT: CONSTRAINT_TWO
- Respond only in: OUTPUT_FORMAT
- Length: LENGTH_BOUND

      Worked example — code review

      A fully-instantiated RISEN prompt, ready to paste into any frontier LLM. No placeholder tokens remain.

      ## Role
You are a senior TypeScript engineer with deep expertise in Node.js API design
and security. You communicate at a mid-level engineer audience level.

## Instructions
Review the following TypeScript function and produce a structured audit report.

## Steps
1. Identify all type-safety issues — missing types, any casts, unsafe assertions.
2. Identify all security issues — injection vectors, unvalidated inputs, secrets
   in scope.
3. Identify all logic errors — off-by-one, null dereference, unhandled rejections.
4. For each finding, state: severity (CRITICAL | HIGH | MEDIUM | LOW), the
   exact line or pattern, and a one-sentence fix.
5. If ambiguity is encountered at any step, state the assumption you are making
   before proceeding.

## End-goal
The output is successful when every finding has a severity label, a code
location, and an actionable fix. Zero findings is a valid result if the code
is clean.

## Narrowing
- Do NOT rewrite the function or produce corrected code.
- Do NOT speculate about intent beyond what the code states.
- Respond only in: markdown, using a table with columns
  Severity | Location | Issue | Fix.
- Length: no preamble, no summary — table only.

      Token cost & when to specialise

      • Token cost is low. The RISEN scaffold adds ~80–120 tokens of overhead per call. No reasoning trace is generated. This makes it the default choice before reaching for chain-of-thought.
      • Add few-shot examples when format compliance is critical. If downstream parsing depends on exact structural output (e.g. a specific JSON schema), add 2–3 concrete input/output pairs after the Narrowing block. Few-shot’s role on frontier models is format alignment, not reasoning improvement (Vellum, 2026; Tianpan, 2026).
      • Add Chain-of-Thought when the task is multi-step reasoning. If Steps 1–4 involve arithmetic, logical deduction chains, or planning with dependencies, layer the CoT skill on top of this scaffold rather than extending it ad-hoc.
      • Re-test after model upgrades. Prompting strategy effectiveness is not stable across model versions. Instruction-following improvements in each generation reduce the marginal value of structural scaffolding. Re-benchmark quarterly (Tianpan, 2026).
